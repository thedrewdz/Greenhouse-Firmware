# MQTT Topic Architecture

# Overview

MQTT is the central communication backbone of the greenhouse platform.

Topic naming should remain:
- hierarchical
- predictable
- human-readable
- scalable

Avoid redundancy where possible to keep things simple.

---

# Topic Design Principles

## Consistency

All devices follow consistent naming conventions.

---

## Readability

Topics should remain understandable during debugging.

---

## Scalability

Topic hierarchy must support:
- multiple devices
- multiple greenhouses (each has its own control unit with paired sensor and actuator units, but could live on the same WiFi)
- future expansion

---

# Proposed Topic Structure

## Device Identification

Use WiFi MAC address as a unit's device ID
- Format: {mac_address}
- In Phase 1, no long-term configuration is stored on peripheral units
- Allows main unit to maintain peripheral awareness and config.

## Command Topics

Issue an instruction to read/write to a peripheral

### Telemetry

`ghcmd/{commandType}-{deviceId}`

- `ghcmd` - This is a command topic
- `{commandType}` - `[rd|wr]`, `rd` - Read, `wr` - Write.
- `{deviceId}` - Each device listens to an instruction topic on its own device ID derived from its WiFi MAC address
- The payload body contains the instruction
- Read instructions could request values from a specific sensor, or the unit's current status.
- Read and write commands are on separate topics for clear intent and simple topic filtering.
- The topic format remains the same while payload instructions can differ by command.

Topic Examples:

```
ghcmd/rd-1ADD5912AF61
ghcmd/rd-3555FA1BD1EE
ghcmd/wr-F11234AABC1A
ghcmd/wr-3555FA1BD1EE
```

**Read Command Example:**

Requests the value of the sensor on slot 0.

```json
{
  "id": 501,
  "slot_id": 0,
  "state": "none",
  "value": 0
}
```

**Write Command Example:**

Instructs the peripheral unit to turn the actuator on slot 4 on (high).

```json
{
  "id": 502,
  "slot_id": 4,
  "state": "on",
  "value": 1
}
```

## Command Payload Contract (Phase 1)

All commands use a shared schema to simplify firmware and control-unit parsing.

```json
{
  "id": 502,
  "slot_id": 4,
  "state": "on",
  "value": 1
}
```

Field requirements and types:

- id: required integer, monotonic message index generated by main control unit.
- slot_id: required integer, target slot index on peripheral control unit.
- state: required string, use none for read commands; use values such as on or off for write commands.
- value: required number, use 0 as default when command does not need a numeric value.

This contract allows one parser for both read and write topics.

## Response Topics

Peripheral unit publishes a response payload in response to a command.

### Telemetry

`gh/{responseType}`

- `gh` - This is a response topic namespace
- `{responseType}` - `[ack|rd|heartbeat]` - `ack` - Acknowledge receipt of write command with result. `rd` - Reading from a sensor. `heartbeat` - Special case, see below.
- Acknowledge and readings use the same response schema in Phase 1.

**Acknowledge Message:**

Reports that the relay on slot 4 has been activated.

```json
{
  "id": 123,
  "device_id": "F11234AABC1A",
  "slot_id": 4,
  "value": 0,
  "state": "on",
  "error_code": 0
}
```

**Read Response Message:**

Returns the temperature in Celsius from the temperature sensor on slot 5

```json
{
  "id": 235,
  "device_id": "1ADD5912AF61",
  "slot_id": 5,
  "value": 19.2,
  "state": "none",
  "error_code": 0
}
```

## Response Payload Contract (Phase 1)

All responses from peripheral control units use a shared canonical schema.

```json
{
  "id": 235,
  "device_id": "1ADD5912AF61",
  "slot_id": 5,
  "value": 19.2,
  "state": "none",
  "error_code": 0
}
```

Field requirements and types:

- id: required integer, peripheral-generated message index.
- device_id: required string, peripheral WiFi MAC address.
- slot_id: required integer, source or target slot index.
- value: required number, canonicalized value or 0 default.
- state: required string, canonical state or none.
- error_code: required integer, 0 for success, non-zero for error.

**Error Response**

- Commands can result in an error
- especially if a user misconfigures a unit during setup. 
- Use a minimum stable Phase 1 error code set:
  - `0` = `none`
  - `1000` = `unknown_command`
  - `1001` = `invalid_payload`
  - `1002` = `invalid_slot`
  - `1003` = `unsupported_capability`
  - `1004` = `invalid_state`
  - `1005` = `device_fault`

**Error Example:**

Reports that the unit cannot comply with the request.

```json
{
  "id": 123,
  "device_id": "F11234AABC1A",
  "slot_id": 4,
  "value": 0,
  "state": "error",
  "error_code": 1002
}
```

## Heartbeat

- Peripheral units publish a heartbeat on a regular interval
- Also used as `is_online` status

`gh/heartbeat`

Example Payload:

```json
{
  "id": 8339,
  "device_id": "1ADD5912AF61",
  "hardware_revision": "A",
  "firmware_version": "1.0.3",
  "uptime_seconds": 92384,
  "wifi_rssi": -61,
  "slot_count": 8,
  "slots": [
    {
      "slot_id": 0,
      "direction": "sensor",
      "i2c_address": "0x25",
      "capability": "moisture",
      "state": "none",
      "value": 41.7,
      "error_code": 0
    },
    {
      "slot_id": 4,
      "direction": "actuator",
      "i2c_address": "0x51",
      "capability": "pump",
      "state": "off",
      "value": 0,
      "error_code": 0
    }
  ],
  "capabilities": [
    "moisture",
    "pump"
  ]
}
```

Heartbeat field requirements and types:

- id: required integer, peripheral message index.
- device_id: required string, peripheral WiFi MAC address.
- hardware_revision: required string.
- firmware_version: required string.
- uptime_seconds: required integer.
- wifi_rssi: required integer.
- slot_count: required integer.
- slots: required array of slot state objects discovered at startup and updated during operation.
- capabilities: required array of canonical capability names.

Per-slot object requirements:

- slot_id: required integer.
- direction: required string, sensor or actuator.
- i2c_address: required string, hex address such as 0x25.
- capability: required string, canonical capability name.
- state: required string, canonical state or none.
- value: required number, canonical value or 0 default.
- error_code: required integer, 0 for success.

Main unit processing requirements:

- Treat each heartbeat as liveness plus topology/state update.
- If device configuration does not exist, trigger onboarding flow.
- If slot topology or module identity differs from stored configuration, trigger reconfiguration flow.
- If topology matches, update runtime state only.

`ghota/{deviceId}/update`

## OTA Status

`ghota/{deviceId}/status`

# Payload Format

Preferred payload format:

- JSON in Phase 1 using `cJSON` on ESP32 firmware

JSON is preferred initially for:

- debugging
- readability
- development speed

Firmware implementation rule:

- Keep a dedicated codec module that maps MQTT payloads to canonical internal message models.
- Keep JSON parse and serialization in the codec module only.
- Keep transport code (`esp-mqtt`) independent from payload parsing logic.

This transport and codec separation preserves flexibility for future payload format changes.

## Phase 2 Format Evaluation

If resource or reliability constraints appear in production-like testing, evaluate a compact binary format (for example CBOR or Protobuf) against JSON.

A format pivot should be data-driven using measured:

- payload size and network usage
- parse and serialize CPU cost
- memory allocation behavior
- end-to-end reliability at target message rates

Payload naming convention:

- Use snake_case for MQTT JSON payload fields.

# Retained Messages

Recommended retained topics:

- device status
- actuator state
- configuration

Avoid retaining:

- high-frequency telemetry (heartbeats)

# Topic Naming Recommendations

- Use lowercase
- hyphens
- not underscores

This naming recommendation applies to topic names, not JSON payload field names.

Avoid:

- spaces
- special characters

## General Naming Considerations

- Proper design means backward compatibility even after devices are deployed
- Keep number of topics low - avoid redundancy
- Peripheral subscribes to: `ghcmd/rd-{deviceId}` and `ghcmd/wr-{deviceId}`
- Peripheral publishes to: `gh/heartbeat`, `gh/ack`, `gh/rd`
- Main Unit Subscribes to: `gh/heartbeat`, `gh/ack`, `gh/rd`
- Main Unit publishes to: `ghcmd/rd-{deviceId}` and `ghcmd/wr-{deviceId}`
